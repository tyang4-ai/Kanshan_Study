我是川大生物医学工程本科在读，最近报了知乎 2026 黑客松。一个人组队，从 4 月下旬开始用 Claude Code 干，到 5 月 10 号写这篇的时候已经写了大概 14 个 phase，1000+ 测试通过。整个过程跟我之前写课程作业完全是两回事。

最大的差异是：我不再是「写代码的人」，我是「定方向的人」。

举个具体的。前两天要给 9 只狐狸 agent 各搭一套 prompt，按之前的写法，我会自己琢磨 prompt 怎么写、出了 bug 一行行调。这次我直接说「按 spec 写 9 个 prompt，每个 200 字以内，要 JSON-strict 输出」，10 分钟后 Claude 把 prompt 给我，我读了一遍，挑了 2 个想改的地方说「这两个不够具体」，再 5 分钟就调好了。整个过程我没看一眼实际的 prompt 模板代码。

这听起来像 vibe coding，但其实有个隐藏的认知负担：**你必须知道什么时候该停下来读代码**。前几天有一次，Claude 给我加了一个「自动重试 + 退避」的逻辑，单元测试都过了，我差点直接 commit。最后多看了一眼，发现它在 401 鉴权失败的时候也会重试——这等于是在用我的 API quota 撞墙。我让它去掉了。如果我那天偷懒不读 diff，这个 bug 至少要等到生产环境 burst 几百次才会被发现。

所以本科生用 AI 写代码不是「省事」，而是「换了一种忙法」。我现在每天的时间分配大概是：
- 30% 想清楚下一步要什么
- 30% 读 AI 写出来的 diff 能不能上
- 25% 跑测试 + 看浏览器有没有炸
- 15% 跟 Claude 来回对话拍方向

写代码本身从我日程表上几乎消失了。

但有个问题没那么好回答：**这种工作方式在毕业以后有没有未来？** 我不知道。我能想到两个方向。一个是这就是新常态，到 2027 年大家都这么干，谁还纠结手敲就是错过了一辆车。另一个是这只是个甜蜜窗口期——等所有应届生都会用 Claude Code，那「会用 AI」就不是技能了，「能想清楚要什么」才是真正的技能，而后者跟会不会用 AI 没关系。

我个人押注是后者。这也是我做看山书房的隐藏动机：如果"会写代码"会贬值，那"用得稳定的多 agent 工作流"就值钱。

这周末要把生物医学工程的概率论作业交了，我打算继续手写。AI 可以帮我做选择题——但概率论的解题思路不像写 React 组件那样有标准模板，AI 给的步骤经常对一半错一半，自己理一遍反而快。

工具的边界比我以为的清楚。
